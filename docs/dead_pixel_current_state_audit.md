## Implemented Now

### Core engine and run flow

- SDL2 window creation, OpenGL rendering, GLEW setup, shader loading, sprite batching, bitmap font rendering, sprite sheet loading, and audio initialization are in place.
- The game runs on a fixed-step update loop with render interpolation.
- There is a functioning ECS with entity IDs, dense component arrays, component registration, entity destruction, and world iteration.
- The main run states already exist: title screen, playing, paused, dead, and retired.
- Title screen, pause menu, death screen, and retirement screen are all active in the run loop.

### Camera, world rendering, and map layer

- The game renders in an isometric projection with zoomable camera controls.
- Camera follow, snap, clamp-to-bounds, and screen/world conversion are implemented.
- Tile rendering is active and sorted back-to-front for isometric display.
- Runtime world generation is active through `MapGenerator`, producing roads, buildings, interiors, decor, indoor loot spawn points, outdoor spawn points, and zombie spawn points.
- Tile collision and surface-type queries are implemented.
- A* pathfinding is implemented and used by zombies during chase behavior.

### Player movement and control

- Isometric WASD movement is implemented.
- Mouse-facing direction is implemented.
- Crouch and run states are implemented.
- Movement speed is affected by stamina and traits.
- Camera follows the player correctly through the current map.

### Needs and survival simulation

- Hunger, thirst, fatigue, sleep debt, body temperature, and illness are all implemented as real runtime values.
- Hunger and thirst damage the player when empty.
- Fatigue and sleep debt lower stamina ceiling.
- Night lowers body temperature and can increase illness.
- Campfires warm the player when nearby.
- Food spoilage exists and affects item condition over time.
- Sleep spots are usable, sleep fast-forwards time, and sleep can be interrupted by nearby zombies.
- Forced collapse from maxed sleep debt exists.

### Combat and stamina

- Melee combat has windup, active, and recovery phases.
- Attack arc visualization is rendered during windup.
- Stamina cost, stamina recovery delay, exhausted attacks, and reduced performance at low stamina are implemented.
- Multiple weapon categories exist in code: improvised, bladed, blunt, ranged, and unarmed.
- Weapon durability and condition degradation are implemented.
- Right-click grab, hold, and execution flow exists.
- Projectiles exist for ranged weapons.
- Combat emits noise and can spawn blood particles.

### Zombies and threat response

- Zombies wander, investigate noise, chase on sight, and attack in melee.
- Zombie line-of-sight checks are implemented.
- Zombie repathing uses A*.
- Zombies attack the player and can inflict bites, cuts, and bruises.
- Zombies react to noise events generated by footsteps, combat, campfires, and other systems.
- Night increases zombie vision range.

### Noise model

- The game has an event-driven noise system with active noise events, decay, tier-based radius, and debug visualization.
- Noise tiers exist from `Whisper` through `Explosive`.
- Footstep noise uses running, crouching, and surface modifiers.
- Noise debug overlay exists and can be toggled.

### Inventory, loot, and item use

- There is a 20-slot inventory with weight limit, stacking, active hotbar slot, and per-slot condition.
- Hotbar selection is active on keys `1` through `8`.
- Inventory overlay exists and supports cursor navigation plus swapping active hotbar items.
- Ground pickups and container loot work.
- Medical caches and supply caches exist.
- Food, drink, medical items, weapons, and materials are defined in `assets/data/items.json` and loaded into the game.
- Using food, drink, medicine, and weapons from inventory is implemented.

### Medical, wounds, and health

- Bleeding damages over time and can be stopped.
- Wounds are tracked separately from raw HP.
- Wound types include bite, cut, and bruise.
- Bites can become infected.
- Infected wounds increase illness.
- Wounds reduce max HP.
- Bandages, antibiotics, painkillers, and first aid kits have active gameplay effects.

### Mental state and retirement

- Panic, depression, and desensitisation are all active runtime systems.
- Panic can increase from zombie proximity, darkness, low HP, and low-needs pressure.
- High panic can emit a scream noise event.
- Panic can add movement drift.
- Depression slows movement.
- Desensitisation rises from zombie kills.
- Hitting maximum depression triggers retirement instead of death.

### Building and structure persistence

- Build mode exists and is enterable in-game.
- Current buildable structures are barricade, noise trap, campfire, sleeping area, supply cache, and rain collector.
- Structures decay over time.
- Barricades block movement and take damage from nearby zombies.
- Noise traps trigger when zombies step on them.
- Campfires emit periodic noise.
- Rain collectors generate water into their container inventory.
- Structures are saved on death/retirement and restored on the next run in a degraded state.

### Legacy, saving, and meta progression

- Save slot metadata exists for world seed, run number, total kills, longest run, and total days survived.
- Grave records persist between runs.
- Graves are spawned back into the world on later runs.
- Each grave also spawns a nearby flavor-note journal entry.
- Structure persistence exists with degradation across runs.
- Save writes use a temp file plus rename flow and maintain backup copies.

### UI, presentation, particles, and audio

- The game already has visible HUD bars for needs.
- Text overlays exist for health, stamina, day/time, character name, traits, objectives, run stats, and interaction messages.
- Inventory and build overlays are functional.
- Day/night lighting is applied through shader uniforms.
- Particle effects exist for blood, dust, embers, and healing.
- Audio assets exist and are loaded for footsteps, combat, zombie sounds, eating, death, hurt, and ambient wind.
- Stereo panning exists for positional sound playback.

## Implemented But Rough / Partial

### World structure and persistence are not aligned with the GDD

- The GDD requires one fixed world layout per save slot.
- The current game regenerates the map using `currentRunSeed()`, which is `worldSeed ^ runNumber`.
- Result: layout changes between runs instead of staying spatially consistent.
- This is the biggest design mismatch in the current codebase.

### Districts do not exist yet

- The current map is a generic procedural town with roads and buildings.
- There are no residential, commercial, industrial, medical, or forest districts.
- There is no district-aware loot logic, infection timeline, or campaign placement logic.

### Tiled support exists but is not the active path

- `TiledLoader` exists.
- `assets/maps/test_map.json` exists.
- The active game path does not use the Tiled loader; the world is generated in code by `MapGenerator`.
- This should either be removed, reactivated, or explicitly kept as tooling/scaffolding.

### Combat is ahead of content

- Ranged/projectile code exists.
- Item data includes ammo.
- The current world setup does not look like a finished ranged-weapon loop yet.
- There is no full firearm content path comparable to melee.
- There are no throwables, shove, knockdown, or blade-stuck behaviors from the GDD.

### Noise is working, but still simplified

- Surface modifiers are active.
- The debug overlay is useful.
- Door/wall occlusion, wind direction, basement isolation, and fog-related perception changes are not implemented.
- Current tier radii are closer to prototype tuning than final GDD values.

### Sleep exists, but not at final design depth

- Sleep spots and interruption work.
- Collapse from sleep deprivation works.
- The full unsafe/marginal/safe/optimal sleep-quality model from the GDD is not implemented.
- Noise traps do not yet create the full sleep-defense tradeoff described in the document.

### Mental health is present, but simplified

- Panic, depression, and retirement exist.
- Retirement currently happens when depression reaches `100`.
- The GDD requires a 24-hour crisis window before psychological retirement.
- There is no human-contact recovery loop because NPC survivors do not exist yet.
- There is no Hardened legacy unlock path.

### Legacy is only partly built

- Graves and degraded structures persist.
- Journal flavor notes exist near graves.
- Bloodline traits, notebook persistence, world memory for burned/looted areas, retirement markers, and broader legacy carryover are not implemented.

### Save system is solid for metadata, but not complete versus the GDD

- Meta, legacy, and structure saves exist.
- Atomic writes and backup copies are already implemented.
- There is no separate run-state checkpoint/autosave system.
- There is no save-slot selection UI; the code always loads `default_slot`.

### UI is functional, but still prototype-grade

- The HUD is readable and useful.
- The current UI is not yet the full top-left/top-center/top-right/bottom-strip interface described in the GDD.
- There is no minimap, no world map, no notebook UI, and no NPC interaction UI.
- Interaction is mostly message-based, not context-panel based.

### Building is present, but still a subset

- The current build list is useful for a survival prototype.
- Metal barricades, door braces, workbenches, fire traps, generators, repairs, and dismantling/recovery loops are not implemented.
- Rain collectors currently generate water without any real weather system behind them.

### Visual and content assets are still placeholder-heavy

- The runtime map tiles still come from `assets/sprites/placeholder.png`.
- A full sprite sheet also exists, but the map layer is still using placeholder art.
- This is fine for prototyping, but it is a clear polish blocker.

### Documentation is out of sync

- `docs/dead_pixel_survival_gdd_v0.3.md` is the target spec.
- `docs/mvl_implementation_plan.md` still says the project has no code and is now outdated.
- `docs/dead_pixel_survival_gdd.html` currently has an empty CSS ruleset lint issue.

## Missing From The GDD

- Weather, seasons, fog, storms, blizzards, heatwaves, and environmental forecasting.
- District infection spread, overwhelmed districts, adjacency-based spread, and district-level simulation.

- Crafting discovery system and notebook-based recipe memory.

- Campaign mode, objective placement algorithm, objective infection timeline, and win condition.

- Tone presets.
- Character creation and manual trait selection.
- Skill XP, skill decay, and skill-gated discovery/progression systems.
- Minimap, full-screen world map, notebook/journal screen, and contextual NPC interaction panels.

- Broader world memory for burned areas, permanently looted districts, and persistent NPC consequences.

## What Needs To Be Fixed Or Polished First

### Priority 0 - Design/architecture fixes

- Fix the world persistence model so one save slot keeps one stable layout.
- Split world-layout persistence from per-run variance.
- Decide whether the game is generator-first or Tiled-first and remove the ambiguity.
- Separate death legacy from retirement legacy instead of storing retirement as a normal grave.

### Priority 1 - Stabilize the current playable prototype

- Add a real run-state autosave/checkpoint system if the GDD requirement still stands.
- Surface contextual interaction prompts so the player does not need to guess when `E` will work.
- Finish the current ranged-weapon path or temporarily hide it from the playable build.
- Replace placeholder map art with a proper temporary production tileset.
- Rebalance early-game survival pressure; the current save snapshot shows most runs ending on Day 1.
- Add basic repair/dismantle loops for built structures.

### Priority 2 - UI and usability polish

- Add a minimap or temporarily add a debug map overlay for navigation.
- Bring the HUD closer to the GDD layout instead of relying mostly on text overlays.
- Improve container looting UX beyond the current one-key transfer model.
- Add clearer feedback for encumbrance, panic, depression, infection, and sleep danger.
- Clean up the outdated docs and fix the GDD HTML lint issue.

### Priority 3 - Big missing systems

- Add district abstraction before attempting infection spread or campaign logic.
- Add NPC survivors before trying to complete the mental-health design or survivor economy.
- Add weather/seasons before finishing body temperature and environment-driven noise design.
- Add crafting/notebook systems before expanding deeper base-building or comfort loops.
- Add vehicles and fire only after the world structure and save architecture are stable.

## Suggested Build Order From Here

1. Lock the save-slot world layout so legacy actually means something.
2. Clean up prototype inconsistencies: prompts, autosave decision, ranged path, placeholder map art, stale docs.
3. Add district abstraction and district-aware data.
4. Build NPC survivors and the first pass of social simulation.
5. Add weather, infection spread, and the broader world-aging model.
6. Add crafting/notebook systems, then campaign/threat modes, then vehicles/fire.

## Bottom Line

Dead Pixel Survival is no longer just an MVL scaffold. It is already a playable survival prototype with a real update loop, real combat, real needs, real death/retirement states, real saves, and real persistence hooks.

The current gap is not “there is no game.” The current gap is that the project has reached the point where architecture choices now matter more than adding isolated features. The fixed-world legacy model, district abstraction, and missing higher-layer systems need to be resolved before the rest of the GDD can land cleanly.
