// game design document — v0.1 brainstorm
DEAD PIXEL
SURVIVAL
A PZ-inspired pixel art survival game — solo C++ project
C++ / SDL2 / OpenGL Pixel Art Procedural Realistic Simulation Solo Dev Desktop
// concept

Dead Pixel Survival is a top-down pixel art zombie survival game built from scratch in C++. Inspired by Project Zomboid's deep simulation philosophy, it adds procedural world generation, a legacy system between deaths, and player-defined threat modes — making every run feel distinct while preserving continuity across deaths.

// four design pillars
Every action has weight
Noise, light, and smell attract zombies. Resource decisions matter. No safe zones last forever.
Slow burn tension
Resources deplete. Infection spreads. Time is always working against the player.
Emergent stories
Traits, events, and legacy systems create organic narratives no two players share.
Pixel-first identity
Art constraint is a feature. Chunky tiles, readable sprites, moody palettes define the look.

// map generation
approach Procedural layout, handcrafted assets. The generator assembles a town from pre-built building templates. Roads, districts, and placement are random — buildings are hand-designed pixel art tiles.
why Fully procedural maps feel incoherent. Handcrafted templates give variety per run while keeping the world believable and the art scope manageable for a solo dev.
districts Residential, commercial, industrial, medical, forest outskirts — each with distinct loot tables, zombie density, and threat profiles.
run variance Building placement, loot seeding, zombie spawn density, and weather conditions all randomize per run. Same templates, different world every time.
// world feel
Day / night cycle
Night dramatically increases zombie aggression and spawn rate. Light sources become critical and dangerous simultaneously.
Weather system
Rain masks sound (stealth benefit) but reduces visibility and accelerates illness. Heat affects stamina drain.
World decay
The longer a run goes, the more the world degrades. Power fails, water stops, food spoils faster.
// threat modes — player selects at run start
// 01
The Overwhelming Swarm
Individually weak but endless. One zombie is manageable — fifty is a death sentence. Panic and resource anxiety.
// 02
The Stalker
Few zombies, but persistent. They hear, track, and never give up. Pure paranoia and stealth gameplay.
// 03
The Mutating Threat
Zombies evolve as days pass. Early: slow and dumb. Late: faster, climb obstacles, break doors. A ticking clock.
// 04
The Infected Society
Survivors are equally dangerous. Factions, desperation, unpredictable NPCs. Zombies are background noise.
// 05
The Unknown
Zombies behave inconsistently — sometimes passive, sometimes frenzied. Triggers are mysteries to discover.
// 06
Hybrid: Swarm + Mutation
Starts as overwhelming swarm. Over days, they mutate into something worse. Two-phase escalating threat.
// legacy system — all five active simultaneously, or toggled per preference
✍
Journal Entries
Dead characters leave notes scattered in the world. Future runs can find them — warnings, locations, emotional artifacts from past lives.
⌂
Persistent Structures
Bases partially survive into the next run in a decayed state. Barricades crumble, supplies deplete, but the bones of your work remain.
⚙
Bloodline Traits
Each death unlocks a trait or perk available to future characters. Die as a skilled carpenter — the next character starts with a fragment of that knowledge.
◎
World Memory
The world remembers what happened. A town you burned stays burned. A horde you drew to the east stays there. Actions have permanent consequences.
†
Named Graves
Every dead character gets a grave in the world, findable by future runs. A growing cemetery that tells the story of all your failures.

Legacy depth is a player preference — toggle individual features or use presets: Fresh Start (nothing carries over) or Living World (all five active).

// primary game modes
Campaign Mode
A goal exists — escape the city, reach a radio signal, find a cure. The procedural map has a destination baked in. Each failed run is a chapter, not the end. Legacy system carries weight here — you die closer to knowing the world.
narrative wrapper win condition legacy-heavy
Endless Mode
No win condition. Pure survival sandbox. How long can you last? The world gets richer the more you die — legacy builds a living history over time. Death is expected. The world outlives the character.
no win condition sandbox legacy as history
Shared engine, two lenses
Both modes run on the exact same systems — same map generator, survival mechanics, zombie behavior, legacy features. Campaign adds an objective layer and narrative wrapper. You're building one game with two playstyles, not two games.
// tone presets — atmosphere per run
Hopeless
PZ-faithful. Bleak, sparse UI, no music. The world is ending and you know it.
Grim but human
Dark but with warmth. Found notes, survivor stories, small victories that matter.
Brutal arcade
Systems stay realistic but the tone is faster, more aggressive. Action-horror energy.
// core simulation systems
needs Hunger, thirst, fatigue, morale, body temperature, illness — displayed as status indicators, not raw numbers. Decay rates affected by weather, activity, and injury.
skills XP earned through actions — carpentry from building, foraging from scavenging, combat from fighting. Skills degrade without use over long runs.
traits Character creation with positive/negative tradeoffs — Fast Learner costs a trait slot, Smoker adds a periodic need. Traits shape playstyle from run start.
loot Container-based scavenging — backpacks, shelves, vehicles, bodies. Loot tables per district. Items degrade, food spoils, weapons break.
base building Fortify locations with barricades, noise traps, supply caches. Structures degrade over time. Built things persist partially into next runs via legacy.
zombie AI Sound and light attracted. Memory-based pathing — they investigate last known position. Horde density increases at night. Behavior shifts per threat mode.
injury system Wounds require treatment — bandaging, disinfecting, rest. Untreated wounds cause infection. Broken bones affect mobility. Illness from exposure or bites.
// tech stack
language C++ — raw performance, direct memory control, no GC pauses. Industry standard for game engines.
windowing SDL2 — handles window creation, input, and rendering surface. Cross-platform, minimal overhead.
rendering OpenGL — GPU rendering for sprites and tilemaps. More approachable than Vulkan for a first custom engine.
audio SDL_mixer — sound effects and ambient audio. Lightweight, integrates naturally with SDL2.
image loading stb_image — single-header library for loading PNG sprite sheets.
map tooling Tiled + custom C++ loader — design building templates in Tiled, load and assemble them procedurally at runtime.
save system nlohmann/json — save files, legacy world state, run history stored as JSON.
art pipeline Aseprite — pixel art and animation. Export sprite sheets directly into the game's asset folder.
architecture Entity Component System (ECS) — entities are IDs, components are pure data, systems contain logic. Scales well for simulation-heavy games.
build system CMake — cross-platform build configuration. Standard for C++ projects.
// exploration roadmap — solo, no pressure
  • Month 1
    C++ foundations + SDL2 window
    Learn pointers, memory management, headers, CMake. Get a window open, a sprite rendering on screen, and basic keyboard input working. Feels slow — this is the most important phase.
  • Month 2
    Core game loop + ECS architecture
    Build your Entity Component System from scratch. This is the backbone everything else plugs into. Get a character moving with basic physics and collision.
  • Month 3
    Tile map + camera system
    Load a Tiled map, render it with OpenGL, implement a camera that follows the player. At this point it starts feeling like a game. First real milestone worth showing.
  • Month 4
    Survival systems v1
    Hunger, thirst, fatigue. Inventory system. Basic loot containers. Day/night cycle. The game starts having tension even without zombies.
  • Month 5
    Zombie AI + threat system
    Basic zombie entities with sound-attraction pathfinding (A*). Implement one threat mode fully (Swarm). This is where it becomes a zombie game.
  • Month 6+
    Procedural map generator + legacy system
    Assemble building templates procedurally. Implement save/load and the first legacy features — named graves and world memory. From here, the game has a soul.

This is an exploration project — timelines are guidelines, not deadlines. Go as deep as you want on any system. Rebuild things when you understand them better. That's the point.
